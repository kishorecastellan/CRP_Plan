Introduction to Cursor

Cursor is a next-generation AI-powered code editor built on VS Code, designed to dramatically enhance developer productivity through intelligent code completion, contextual assistance, and AI-driven code generation.

üéØ Why Cursor for Experienced Developers?

As a seasoned AWS Full Stack Developer, Cursor empowers you to:

  • Accelerate Development: Generate boilerplate code, complex functions, and entire modules in seconds
  • Reduce Context Switching: Get answers, documentation, and code suggestions without leaving your editor
  • Maintain Code Quality: AI-assisted code review, refactoring, and best practice suggestions
  • Learn New Technologies: Quickly understand unfamiliar codebases and frameworks
  • Automate Repetitive Tasks: Let AI handle routine coding tasks while you focus on architecture

üöÄ Cursor's Core Philosophy

Cursor treats AI as a pair programmer who understands your entire codebase, not just the current file. This contextual awareness makes it exceptionally powerful for complex, multi-service AWS applications.

Effective Prompting Strategies

The quality of AI assistance depends heavily on how you frame your requests. Master these prompting techniques to get the best results.

üéØ The Anatomy of a Good Prompt

Be Specific About Context

Instead of "fix this code," say "this Lambda function fails when the DynamoDB item doesn't exist - add proper error handling for missing items"

Specify the Technology Stack

Mention the frameworks, libraries, and versions: "using AWS SDK v3 with TypeScript" or "using Python 3.11 with boto3"

Define the Output Format

Tell Cursor what you want: "return a TypeScript function with JSDoc comments" or "include unit tests using Jest"

Include Constraints

Add requirements: "must be compatible with Node.js 18," "should follow our error handling pattern," or "keep it under 50 lines"

üìä Prompt Examples: Bad vs Good

Example 1: Code Generation

‚ùå BAD: "write a function to get user"
‚úÖ GOOD: "Write an async TypeScript function called getUserById that:
- Takes a userId string parameter
- Queries DynamoDB table 'Users' using AWS SDK v3
- Returns the user object or null if not found
- Handles and logs DynamoDB errors appropriately
- Include JSDoc comments"

Example 2: Debugging

‚ùå BAD: "this doesn't work"
‚úÖ GOOD: "This Lambda function throws 'Cannot read property of undefined' 
when processing SQS messages. The error occurs at line 45 when accessing 
message.body.userId. Some messages might not have a body. Help me add 
proper validation and error handling."

Example 3: Refactoring

‚ùå BAD: "make this better"
‚úÖ GOOD: "Refactor this function to:
1. Use async/await instead of callbacks
2. Extract the validation logic into a separate function
3. Add TypeScript types for all parameters and return values
4. Follow our project's error handling pattern using AppError class"

üîÆ Advanced Prompting Techniques

Chain-of-Thought Prompting

Ask Cursor to explain its reasoning for complex problems:

"Analyze this authentication flow step by step, identify potential 
security vulnerabilities, and suggest improvements with explanations 
for each change."

Role-Based Prompting

Set a specific persona for more targeted responses:

"As a senior AWS solutions architect, review this serverless architecture 
and suggest improvements for scalability and cost optimization."

Iterative Refinement

Build on previous responses:

1. "Create a basic Lambda function for user registration"
2. "Now add email validation using a regex pattern"
3. "Add integration with SES to send welcome emails"
4. "Add CloudWatch custom metrics for registration events"

Template Prompting

Use templates for consistent output:

"Generate a Lambda handler following this template:
- Input validation first
- Business logic in try block
- Specific error handling in catch
- Always return proper API Gateway response format
- Include X-Request-Id in response headers"

üß† Context Best Practices

  • Be Selective: Don't overload with context. Include only relevant files to avoid confusing the AI
  • Use @codebase for Discovery: When you're unsure where something is implemented, let AI search
  • Reference Patterns: Point to existing implementations when you want consistent code style
  • Include Types: Always reference type definitions for TypeScript projects
  • Context Window Limits: Be aware that very large contexts may be truncated; prioritize important files

üìã Example Rule Files

General Project Rules (general.mdc)

---
description: General coding standards for the project
globs: *
alwaysApply: true
---

# Project Standards

## Code Style
- Use meaningful variable names (no single letters except loop counters)
- Maximum function length: 50 lines
- Always add JSDoc comments for public functions
- Use early returns to reduce nesting

## Error Handling
- Never swallow errors silently
- Use custom AppError class for application errors
- Log errors with context (requestId, userId, operation)
- Always include stack traces in development

## TypeScript
- Strict mode enabled - no `any` types
- Use interfaces for data structures, types for unions
- Prefer readonly properties where applicable

AWS-Specific Rules (aws.mdc)

---
description: AWS development standards
globs: ["**/handlers/**", "**/services/**"]
alwaysApply: false
---

# AWS Development Standards

## Lambda Functions
- Keep handlers thin - delegate to service layer
- Always set appropriate timeouts
- Use environment variables for configuration
- Implement idempotency for critical operations

## DynamoDB
- Use single-table design where appropriate
- Always specify projection expressions
- Use consistent naming: PK, SK for keys
- Implement exponential backoff for retries

## API Gateway
- Return proper HTTP status codes
- Include request ID in all responses
- Use request validation at API Gateway level
- Implement proper CORS headers

## SDK Usage
- Use AWS SDK v3 with modular imports
- Create clients outside handler for reuse
- Implement proper error handling for SDK calls

Testing Rules (testing.mdc)

---
description: Testing standards
globs: ["**/*.test.ts", "**/*.spec.ts"]
alwaysApply: false
---

# Testing Standards

## Unit Tests
- Use Jest with TypeScript
- Follow AAA pattern (Arrange, Act, Assert)
- One assertion per test when possible
- Mock external dependencies (AWS SDK, APIs)

## Test Naming
- Format: "should [expected behavior] when [condition]"
- Example: "should return null when user not found"

## Mocking
- Use jest.mock() for module mocking
- Create mock factories in __mocks__ directory
- Use aws-sdk-client-mock for AWS SDK v3
